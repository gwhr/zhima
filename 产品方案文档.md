# 智码（ZhiMa）· 产品方案文档

> **核心定位：用户专属的 AI 软件开发者**
> 平台相当于给每个用户配备了一个专业开发者，从选题、开发、到写论文全程包办。
> 所有能力全部由 AI 驱动（选题推荐 / 需求评估 / 代码生成 / 论文生成 / 修改迭代），平台本身不做任何人工介入。
> 用户只需要描述需求、做决定、最终提交。

---

## 一、核心交付物

| 交付物 | 说明 |
|--------|------|
| ① 可运行的项目代码 | 真实可运行，支持在线预览 |
| ② 符合学校格式的论文 | 按模板生成，开箱即用 |
| 后期扩展 | 开题报告、答辩 PPT |

## 二、功能零：选题与需求评估（前置流程）

> 用户完全不懂开发，需要平台引导他明确"要做什么"，并评估可行性，再交给 AI 生成。

### 2.1 选题流程

```
用户输入方向或关键词（可以很模糊）
例如："我想做个外卖的" / "做个学生管理" / "我喜欢游戏"
        ↓
AI 推荐 5~10 个匹配的毕设题目
每个题目附带：
  - 题目名称
  - 一句话介绍
  - 推荐技术栈
  - 难度评级（⭐⭐⭐）
        ↓
用户选一个，或自己输入自定义题目
```

### 2.2 需求展开（核心设计）

用户选题后，AI 自动展开项目的具体开发内容：

```
用户选了"基于Vue3的校园二手交易平台"
        ↓
AI 自动生成项目需求清单：

【系统角色】
  - 普通用户：发布商品、浏览商品、下单购买、聊天沟通
  - 管理员：用户管理、商品审核、订单管理、数据统计

【功能模块】
  ✅ 用户注册/登录（手机号+验证码）
  ✅ 商品发布（图片上传、分类、定价）
  ✅ 商品列表（搜索、筛选、排序）
  ✅ 商品详情页
  ✅ 购物车 & 下单流程
  ✅ 订单管理
  ✅ 用户个人中心
  ✅ 管理后台（数据看板）

【数据库核心表】
  - 用户表、商品表、订单表、分类表、消息表
```

**用户可以自由修改这份清单：**
- 勾选/取消某个功能（"我不需要聊天功能"）
- 补充新功能（"我还想加一个积分系统"）
- 修改角色（"不需要管理员，只要普通用户"）

### 2.3 AI 可行性评估

用户确认需求后，AI 对整个方案进行评估：

```
        ↓ AI 分析 ↓

【复杂度评估】
  整体难度：★★★☆☆（中等）
  
  各模块复杂度：
  - 用户登录注册     ★★☆☆☆ 简单
  - 商品发布（含图片）★★★☆☆ 中等
  - 订单流程         ★★★★☆ 较复杂
  - 管理后台         ★★★☆☆ 中等

【可行性结论】
  ✅ 完全可行，适合作为毕设项目
  ✅ 功能完整，能满足答辩要求
  ✅ 技术栈匹配所选语言

【预计生成内容】
  - 前端页面：约 15 个
  - 后端接口：约 30 个
  - 数据库表：约 8 张
  - 论文字数：约 15000 字

【风险提示】
  ⚠️ 图片上传功能需要配置存储服务
  ⚠️ 订单模块逻辑较复杂，如遇问题可简化为"直接购买"
```

### 2.4 用户确认 → 进入生成

```
用户看到评估结果
  → 满意：点「开始生成」→ 进入代码生成流程
  → 想简化：去掉几个功能 → 重新评估
  → 想加功能：补充需求 → 重新评估
        ↓
最终确认的需求清单直接传给 AI
作为代码生成和论文生成的统一输入
```

### 2.5 设计价值

```
对用户：
  不懂开发也能清楚知道自己要做什么
  提前知道复杂度，不会做到一半发现做不了
  需求和代码、论文完全一致，不会"代码是一个，论文写另一个"

对平台：
  需求清单是整个工作空间的"锚点"
  代码生成、论文生成都基于同一份需求
  后续修改也有明确的上下文依据
```

## 三、功能一：项目代码生成

### 2.1 用户选择技术栈

**后端语言（三选一）：**
- Node.js（Express / Koa）
- Java（Spring Boot）
- Python（FastAPI / Flask）

**数据库（自由选择）：**
- MySQL（默认推荐）
- PostgreSQL
- MongoDB

**前端框架（自由选择）：**
- Vue3（默认）
- React

### 2.2 代码生成流程

```
用户输入题目 + 选择技术栈
        ↓
调用 AI 接口（DeepSeek / GPT）
生成完整项目代码：
  - 前端页面（完整功能模块）
  - 后端接口（RESTful API）
  - 数据库建表 SQL
  - 演示数据 SQL（种子数据）
  - README（含本地运行步骤）
        ↓
代码保存到用户工作空间
```

### 2.3 在线预览方案

**触发方式：** 用户点击「预览运行」按钮

**运行流程：**
```
服务器拉取对应 Docker 镜像
  镜像A：node:20 + mysql:8
  镜像B：openjdk:17 + mysql:8
  镜像C：python:3.11 + mysql:8
  （按用户选择的技术栈匹配）
        ↓
注入用户项目代码
自动执行建表 SQL + 种子数据 SQL
        ↓
约 30 秒启动完成
返回临时访问链接（含账号密码）
        ↓
用户访问真实运行的网站
可登录、可操作、可截图
        ↓
30 分钟无操作 → 容器自动销毁
再次预览 → 重新启动（数据重置）
```

**用户能看到什么：**
- 真实可交互的前端页面
- 真实后端接口返回数据
- 真实数据库（含 AI 生成的演示数据）
- 账号密码由 AI 提前告知（如 admin / 123456）

### 2.4 迭代修改

用户在工作空间内通过对话持续修改：

```
"帮我加一个用户统计图表"
"登录改成手机号验证码方式"
"管理员角色加权限控制"
        ↓
AI 修改对应代码文件
预览重新启动，效果实时可见
```

对话记录与项目上下文长期保存，随时回来继续。

### 2.5 成本估算

| 项目 | 单次成本 |
|------|---------|
| AI 代码生成 | ≈ 0.5 ~ 1 元 / 次 |
| 容器预览（30分钟） | ≈ 0.05 ~ 0.1 元 / 次 |
| 用户修改一次代码 | ≈ 0.1 ~ 0.3 元 / 次 |

## 五、功能二：论文生成

### 3.1 模板策略

**平台内置标准模板：**
- 我们预先收录各校官方模板，精确调好格式
- 用户选择学校 → 自动匹配对应模板
- 用户下载即是符合学校要求的格式，无需手动调整

**模板扩展策略：**
- 初期内置 3 ~ 5 套主流院校模板
- 用户可提交所在学校模板，平台收录后全校学生共享
- 模板库越大，覆盖越广，形成竞争壁垒

### 3.2 论文生成流程

```
用户工作空间内已有项目代码
        ↓
AI 读取项目结构，自动理解系统功能
（不需要用户重复描述，代码即文档）
        ↓
按学校模板结构生成各章节内容：
  摘要 / Abstract
  第1章 绪论（背景、意义、国内外现状）
  第2章 需求分析（功能需求、用例图描述）
  第3章 系统设计（架构图、数据库设计）
  第4章 系统实现（关键代码说明）
  第5章 系统测试（测试用例、测试结论）
  第6章 结论
  参考文献
  致谢
        ↓
套入学校标准模板（字体/行距/页边距完全正确）
输出 .docx 文件
        ↓
用户在线预览 + 可对话修改
"第三章数据库设计部分写得太简单，帮我补充"
"摘要改成300字以内"
        ↓
满意后一键下载
```

### 3.3 图表处理

论文中的图表由 AI 自动生成：

| 图表类型 | 生成方式 |
|---------|---------|
| 系统架构图 | Mermaid 渲染为图片 |
| 流程图 | Mermaid 渲染为图片 |
| 用例图 | PlantUML 渲染为图片 |
| 数据库 ER 图 | 自动根据建表 SQL 生成 |
| 测试数据图表 | ECharts 渲染为图片 |

### 3.4 论文质量保障

- 论文内容基于用户真实代码生成，内容与代码一致，不是乱编
- 参考文献引用真实文献（接入学术数据库）
- 支持降 AI 味改写（答辩前处理）
- 查重友好：每次生成结果不完全相同

### 3.5 成本估算

| 项目 | 单次成本 |
|------|---------|
| AI 生成全文论文 | ≈ 1 ~ 3 元 / 次 |
| 图表生成 | ≈ 0.1 元 / 套 |
| 修改润色 | ≈ 0.1 ~ 0.5 元 / 次 |

## 九、成本控制机制

> 核心原则：每个用户的 AI 消耗有上限，超额自动降级模型，平台永远不亏钱。

### 9.1 用户成本账户

每个用户创建工作空间时，系统分配一个"成本账户"，每次 AI 调用自动记录消耗。

**各套餐 Opus 额度：**

| 套餐 | 售价 | Opus 额度上限 | 超出后策略 |
|------|------|-------------|-----------|
| 基础版 | ¥49 | ¥25 | 全部切 DeepSeek |
| 标准版 | ¥99 | ¥50 | 全部切 DeepSeek |
| 旗舰版 | ¥199 | ¥100 | 全部切 DeepSeek |

> DeepSeek 调用不计入额度，因为成本可忽略（全程 < ¥1）。

### 9.2 渐进式降级策略

不是突然切换，而是逐步收紧 Opus 使用范围：

```
阶段一（已消耗 0% ~ 60% 额度）
  正常模式
  代码生成 / 图表生成 / 复杂修改 → Opus 4.6
  其他任务 → DeepSeek V3

阶段二（已消耗 60% ~ 90% 额度）
  收紧模式
  仅代码生成 → Opus 4.6
  图表 / 复杂修改 → 降级为 DeepSeek V3
  其他任务 → DeepSeek V3

阶段三（已消耗 90% ~ 100% 额度）
  经济模式
  全部任务 → DeepSeek V3

超出 100% 额度
  → 提示用户："修改次数较多，建议升级套餐"
  → 或付费购买额外 Opus 额度（¥10 = ¥10 的 Opus 调用量）
```

### 9.3 用户消耗记录

**数据库表设计：user_token_usage**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint | 主键 |
| user_id | bigint | 用户ID |
| workspace_id | bigint | 工作空间ID |
| task_type | varchar | 任务类型（topic/evaluate/code/thesis/chart/modify） |
| model | varchar | 使用模型（opus-4.6 / deepseek-v3） |
| input_tokens | int | 输入 token 数 |
| output_tokens | int | 输出 token 数 |
| cost_yuan | decimal | 本次消耗金额（元） |
| created_at | datetime | 记录时间 |

**用户维度汇总视图：**

| 字段 | 说明 |
|------|------|
| user_id | 用户ID |
| plan_type | 套餐类型 |
| opus_budget | Opus 额度上限 |
| opus_used | 已消耗 Opus 金额 |
| opus_remaining | Opus 剩余额度 |
| usage_phase | 当前阶段（normal / tightened / economy） |
| total_requests | 总请求次数 |

### 9.4 调用决策流程

```
用户发起 AI 请求
        ↓
判断任务类型（代码生成 / 论文 / 图表 / 修改...）
        ↓
查询用户当前 Opus 消耗阶段

        ↓

        ↓
记录本次 tokens 消耗和费用

更新用户累计消耗
```

| 生成的代码跑不起来 | ⚠️ 中 | 容器预览提前验证，跑不起来自动修复 |
| 论文格式不精确 | ⚠️ 中 | 持续收录和优化各校模板 |
| API 涨价 | ⚠️ 低 | 混合模型策略，不依赖单一供应商 |
| 竞品模仿 | ⚠️ 中 | 先发优势+模板库+用户口碑 |

### 10.4 总体结论


### 25.4 架构设计原则

```
① 单仓库（Monorepo 思想）
   Next.js + Worker 共享 lib/ 目录
   改一次数据库模型，两边自动同步
   不会出现"前后端接口对不上"的问题

② 短任务同步，长任务异步
   < 10秒的任务：API Route 直接处理
   > 10秒的任务：扔进 BullMQ → Worker 处理 → 通知用户

③ AI 调度统一入口
   所有 AI 调用走 lib/ai/router.ts
   根据任务类型 + 用户额度阶段 → 自动选模型
   所有消耗自动记录

④ 数据库是唯一真相源
   任务状态、用户额度、生成进度 → 全在 PostgreSQL
   前端轮询数据库状态展示进度
   不依赖内存状态，服务重启不丢数据
```
