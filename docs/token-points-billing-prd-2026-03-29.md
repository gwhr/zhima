# Token 点数计费 PRD（2026-03-29）

状态：已落地；后续于 2026-04-02 补充了运行时套餐配置与支付回调按配置结算

## 1. 背景

平台不再对用户暴露“模型原生 token 1:1”概念，改为统一售卖“平台 Token（点数）”，避免多模型切换造成亏损或计费口径不一致。

## 2. 目标

1. 用户侧完成“余额-充值-流水”闭环。
2. 生成代码、生成论文、AI 对话统一接入计费链路。
3. 管理侧可配置倍率与汇率，可查流水，可调余额，可做异常回滚。

## 3. 强约束（必须满足）

1. 任务启动先冻结预估点数，结束后按实际结算，多退少补。
2. 冻结/结算/回退必须在数据库事务中完成，防止并发重复扣费。
3. 每次扣费必须记录流水（用户、任务、模型、输入/输出/缓存 token、成本、时间）。
4. 失败任务自动回退冻结额度。
5. 平台必须支持：单任务上限、单日上限、并发上限。
6. 余额不足时禁止任务启动，并给出明确错误提示。

## 4. 计费模型

### 4.1 统一公式

- 模型成本（元）= 输入 token × 输入单价 + 输出 token × 输出单价 + 缓存命中 token × 缓存命中单价
- 扣减点数 = ceil(模型成本 × 计费倍率 × 点数汇率)

### 4.2 口径说明

- 用户看到的是平台点数。
- 平台内部按真实模型消耗核算，支持多模型统一计费。
- 充值包是点数包，不是模型 token 包。

## 5. 用户侧需求

1. 账单页面展示余额（总点数、可用、冻结、已用、今日已扣）。
2. 展示充值包列表并可创建充值订单。
3. 展示个人流水明细（冻结、结算、回退、充值、手动调整等）。

## 6. 管理侧需求

1. 平台配置可调：
  - 计费倍率
  - 点数汇率（点/元）
  - 每日点数上限
  - 代码/论文/对话预冻结点数
2. 模型管理可维护价格表（输入/输出/缓存命中单价）。
3. 用户管理支持：
  - 设置用户默认额度覆盖值
  - 调整钱包总点数（实时生效）
  - 设置用户并发上限
4. Token 流水页支持筛选、查看明细、异常冻结回滚。

## 7. 数据结构

### 7.1 钱包与流水

- `UserTokenWallet`：总点数、可用、冻结、已用、日用量
- `TokenReservation`：冻结记录（ACTIVE/SETTLED/RELEASED）
- `TokenLedger`：流水（INIT_GRANT/RECHARGE/FREEZE/SETTLE/REFUND/MANUAL_ADJUST/ROLLBACK）

### 7.2 任务字段

- `TaskJob.tokenReservedPoints`
- `TaskJob.tokenSettledPoints`
- `TaskJob.tokenCostYuan`
- `TaskJob.tokenSettlementStatus`

## 8. 接口与流程

1. 任务启动（代码/论文/聊天）：
  - 创建任务 -> 冻结点数 -> 入队
2. 任务成功：
  - 读取真实 usage -> 结算点数 -> 记流水
3. 任务失败：
  - 释放冻结 -> 记回退流水
4. 充值回调：
  - 订单成功 -> 充值入账 -> 记充值流水

## 9. 验收标准

1. 三类任务都可看到冻结、结算、回退流水。
2. 后台修改倍率后，新任务扣费立即生效。
3. 钱包余额不足时，任务启动返回明确错误。
4. 管理员手工回滚 ACTIVE 冻结后，用户可用余额恢复。
5. `pnpm test:run` 与 `pnpm build` 通过。
