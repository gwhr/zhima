# PRD：按需求决定端形态 + 代码生成 UI 质量提示（2026-03-31）

## 背景
- 现状里部分提示词默认把项目固定成“用户端 + 管理端”双端，导致与实际需求不完全匹配。
- 代码生成在部分场景会产出“可运行但展示弱”的页面，用户感知为“只有一个页面、UI 太简”。

## 目标
- 让 AI 生成逻辑改为：端形态（单端/双端/多端）由业务需求决定，不强制双端。
- 强化代码生成提示词中的 UI 质量要求，默认输出更可展示的页面结构。

## 本次改动

### 1) 需求扩展接口去“双端强约束”
- 文件：`app/api/ai/expand-requirements/route.ts`
- 调整：
  - 方向约束从“聚焦用户端、管理端”改为“聚焦业务流程、角色权限、数据库、可展示性”。
  - 删除“默认双端系统”硬约束，改为：
    - 按选题决定端形态（单端/双端/多端）。
    - 只有需求明确存在管理职责时才补充管理端模块。
    - 在 `summary` 标明端形态与理由。

### 2) 选题推荐接口去“双端硬要求”
- 文件：`app/api/ai/recommend-topics/route.ts`
- 调整：
  - 题目方向约束改为“可落地为单端或多端，按场景决定”。
  - fallback 文案由“用户端+管理端协同”改为更中性描述，避免先验锁死结构。

### 3) 代码生成提示词增强 UI 质量
- 文件：`worker/index.ts`
- 调整：
  - 规则 7 改为“按需求决定端形态，不强行添加无关端”。
  - 新增 UI 质量规则：
    - 至少 2 个有业务字段/交互的页面。
    - 页面要有基础视觉层次（间距/字号/主次按钮/留白）和桌面端可展示性。
  - 重试纠偏提示同步改造，避免重试时再次强制双端。

### 4) Fallback 骨架去“固定 user/admin 目录”
- 文件：`worker/index.ts`
- 调整：
  - Node/Java fallback API 从 `user/admin` 迁移为中性 `items/stats`。
  - Vue/React fallback 页面改成 `Overview/Workflow/DataBoard`，不再绑定“用户端/管理端”命名。

## 验收标准
- 非管理后台类选题不再被强行生成 admin 页面/路由。
- 管理角色明确时，仍可生成管理模块。
- fallback 项目结构可运行且命名中性。
- 代码生成结果在 UI 展示上不再只有“单页占位文本”。
