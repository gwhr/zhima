# PRD：真实运行预览（5 分钟沙箱 + 排队）可行性评估（2026-03-31）

## 结论（先给结论）
- **可做**，但不是小改动，属于 **P1~P0 之间的大功能**。
- 难点不在“起服务”，而在 **安全隔离 + 资源回收 + 排队调度 + 失败可观测**。

## 为什么当前预览不等同真实运行
- 现有 `preview-build` 是“界面演示页（模拟数据）”，用于快速看信息结构和页面布局。
- 它不启动用户生成项目，不会实际安装依赖/连接数据库/跑服务进程。

## 目标能力（你提的版本）
- 每次点击“运行预览”创建一个临时运行环境。
- 队列排队，页面显示“前方人数/预计等待”。
- 每个用户单次预览最长 5 分钟，到时自动销毁。
- 显示当前占用数量（并发槽位）。

## 推荐技术方案（稳妥版）

### 方案 A（推荐）
- 基础：Docker + 队列（现有 BullMQ 可复用）+ Nginx 反代。
- 流程：
  1. 入队 `PREVIEW_RUNTIME` 任务。
  2. Worker 拉取代码包，构建容器镜像或挂载模板镜像。
  3. 在受限网络/资源下启动容器（CPU/内存/网络白名单）。
  4. 分配临时访问地址（例如 `/runtime/{sessionId}` 反代到容器端口）。
  5. 5 分钟 TTL 到期后强制回收容器、临时卷、代理路由。

### 方案 B（后续升级）
- K8s Job/Pod + Ingress 动态路由，适合规模化并发。
- 比 A 更强，但复杂度和维护成本明显更高。

## 安全与风控必须项
- 禁止特权容器，限制 CPU/内存/进程数。
- 关闭宿主机挂载与 Docker Socket 映射。
- 网络隔离（默认无外网，按白名单放行）。
- 运行用户非 root，文件系统只读 + 临时可写层。
- 统一超时终止 + 异常自动清理（防止“僵尸容器”）。

## 排队与计费建议
- 并发槽位：初期 2~3（2C4G 机器建议从 1~2 开始）。
- 排队展示字段：
  - 前方人数
  - 当前并发占用
  - 预计等待时间（滑动平均）
  - 剩余预览时长
- 计费：可按“预览分钟数”或“预览启动次数”扣平台点数。

## 实施分期
- **Phase 1（1~2 天）**：排队 UI + 占位流程（仍用模拟预览）。
- **Phase 2（3~5 天）**：单语言真实运行（先 Node 或 SpringBoot 二选一）。
- **Phase 3（3~7 天）**：多栈支持 + 自动清理 + 监控告警 + 管理端统计。

## 对当前版本的建议
- 现在先保留“界面演示预览”，避免用户误解为真实运行。
- 等短信、支付、模型稳定后，再上真实运行预览，避免同时引入太多不稳定因素。
