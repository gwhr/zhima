# 管理员端 PD（运营后台）

> 更新时间：2026-03-25
> 适用阶段：MVP 上线后到首个公开测试阶段

## 1. 背景

当前平台的主链路已经聚焦在用户侧：创建工作空间、确认功能、生成代码、生成论文、预览、下载。

但只要开始给真实用户使用，后台就不能只停留在“看用户数和订单数”。
管理员端的核心价值不是展示数据，而是解决下面四件事：

1. 看清平台运行状态：今天有多少用户、多少任务、多少失败、消耗了多少 token。
2. 控制平台成本：哪些用户最耗 token，代码/论文默认模型是什么，额度策略是否合理。
3. 处理异常问题：哪个工作空间生成失败了，哪个任务卡住了，哪个用户需要补额度。
4. 支撑后续收费：订单、配额、模型、模板、公告、风控都要有后台入口。

## 2. 产品目标

管理员端的目标不是“做一个很全的 CRM”，而是先做一个轻量、够用、能控风险的运营后台。

MVP 目标：

1. 能看到用户、工作空间、任务、订单、token 消耗的核心数据。
2. 能调整用户额度和平台默认 AI 配置，避免成本失控。
3. 能定位失败任务和异常用户，快速人工干预。
4. 能为后续付费、模板运营、风控留出扩展位。

## 3. 后台角色

### 3.1 当前阶段

当前只保留一个角色：`ADMIN`

权限范围：

1. 查看全平台用户、工作空间、订单、任务、AI 消耗数据。
2. 修改平台级配置。
3. 调整用户 token 配额。
4. 处理异常任务和系统公告。

### 3.2 后续可扩展

后续如有协作者，再拆成：

1. `SUPER_ADMIN`
   负责模型/key、全局配置、删除数据、管理员管理。
2. `OPERATOR`
   负责用户查询、工单处理、公告发布、额度补发。
3. `FINANCE`
   负责订单、退款、充值记录和财务对账。

## 4. 菜单结构

管理员端建议按下面的导航组织：

1. 概览看板
2. 用户管理
3. 工作空间管理
4. 任务中心
5. Token 与 AI 用量
6. 订单与计费
7. 平台配置
8. 论文模板与素材
9. 系统通知
10. 操作日志

## 5. MVP 功能定义

### 5.1 概览看板

目标：管理员打开后台 30 秒内知道平台是否正常。

首页需要展示：

1. 总用户数
2. 今日新增用户数
3. 工作空间总数
4. 今日新增工作空间数
5. 今日代码生成次数
6. 今日论文生成次数
7. 今日任务失败数
8. 今日 token 消耗总量
9. 今日 AI 成本估算
10. 累计付费订单数和累计收入

建议增加的图表：

1. 最近 7 天用户注册趋势
2. 最近 7 天 token 消耗趋势
3. 最近 7 天任务成功率趋势
4. 模型使用占比
5. 高消耗用户 Top 10

MVP 操作：

1. 点击看板卡片可跳到对应列表页。
2. 支持按时间范围切换：今日 / 近 7 天 / 近 30 天。

### 5.2 用户管理

这部分是后台最重要的页面之一。

列表字段建议：

1. 用户 ID
2. 昵称
3. 邮箱
4. 手机号
5. 角色
6. 注册时间
7. 工作空间数量
8. 订单数量
9. Token 总额度
10. Token 已消耗
11. Token 剩余额度
12. 累计 AI 成本
13. 最近任务状态

详情页建议展示：

1. 基础信息
2. 工作空间列表
3. 最近 20 条 AI 使用记录
4. 最近任务记录
5. 订单记录
6. 通知记录

管理员操作：

1. 搜索用户
2. 查看详情
3. 调整角色
4. 禁用/启用用户
5. 手动补发 token
6. 手动扣减 token
7. 备注异常原因
8. 单独设置该用户并发任务上限（覆盖平台默认值）

为什么必须做：

1. 免费额度阶段，经常会有用户额度不够但需要人工补偿。
2. 如果出现刷 token 或恶意压测，需要能快速封禁。

### 5.3 工作空间管理

用户问题很多最后都要落到工作空间维度排查，所以后台必须有这一页。

列表字段建议：

1. 工作空间 ID
2. 所属用户
3. 项目名称
4. 课题名称
5. 专业大类
6. 当前状态
7. 技术栈
8. 文件数量
9. 已消耗 token
10. 创建时间
11. 更新时间

详情页建议展示：

1. 功能清单快照
2. 难度评估结果
3. 文件列表
4. 最近任务链路
5. 最近 AI 对话
6. 预览构建状态

管理员操作：

1. 查看工作空间详情
2. 查看生成文件
3. 查看失败日志
4. 手动重新触发预览
5. 标记过期
6. 后续支持逻辑删除

### 5.4 任务中心

这页是排查“为什么生成失败”“为什么卡住”的核心入口。

列表字段建议：

1. 任务 ID
2. 任务类型
3. 所属用户
4. 工作空间
5. 当前状态
6. 进度
7. 使用模型
8. 输入 token
9. 输出 token
10. 耗时
11. 错误信息
12. 创建时间

管理员操作：

1. 查看任务详情
2. 查看原始错误信息
3. 重试失败任务
4. 终止长时间卡住任务
5. 按状态筛选：待执行 / 运行中 / 已完成 / 已失败

为什么必须做：

1. 真实上线后，第一类问题就是任务失败和任务卡死。
2. 如果没有任务中心，管理员只能查数据库或看日志，效率很低。

### 5.5 Token 与 AI 用量

这是平台的成本控制中心，也是你提到的重点。

需要展示的数据：

1. 平台总 token 消耗
2. 按模型分组的 token 消耗
3. 按任务类型分组的 token 消耗
4. 按用户分组的 token 消耗
5. 按天的 token 趋势
6. AI 成本估算

列表与图表建议：

1. 用户消耗排行榜
2. 模型消耗占比饼图
3. 代码生成 vs 论文生成 token 对比
4. 高失败高消耗任务列表

管理员操作：

1. 调整默认用户 token 总额度
2. 调整代码生成预留 token
3. 调整论文生成预留 token
4. 给单个用户补 token
5. 导出 token 使用明细

### 5.6 订单与计费

即使第一期先免费，也建议把后台入口留好，因为后面一旦接充值或套餐，会直接用上。

列表字段建议：

1. 订单 ID
2. 用户
3. 套餐类型
4. 金额
5. 状态
6. 支付渠道
7. 第三方单号
8. 创建时间
9. 支付时间

管理员操作：

1. 查询订单
2. 按状态筛选
3. 查看异常订单
4. 后续支持手动补单
5. 后续支持退款登记

### 5.7 平台配置

这一页就是你说的“平台模型 / key 控制”，但要分阶段做。

#### 第一阶段：只做基础可控

管理项：

1. 代码生成默认模型
2. 论文生成默认模型
3. 默认免费 token 额度
4. 代码生成预留 token
5. 论文生成预留 token
6. 是否开启代码生成
7. 是否开启论文生成
8. 是否开启预览构建
9. 是否展示维护公告
10. 默认单用户并发任务上限
11. 同类任务失败重试次数上限
12. 单次任务 token 硬上限（超限中止并提示拆分需求）

说明：

1. 当前模型和 key 仍然可以保留在环境变量中。
2. 后台页面先展示“当前生效配置”和“连通状态”，不强行在页面上直接改 secret。
3. 这样上线风险更低，也更符合当前阶段。

#### 第二阶段：真正的模型/key 池

后续再支持：

1. 多供应商配置
2. 多 key 池
3. key 启用/停用
4. key 失败次数统计
5. 自动切换与降级策略
6. 模型价格配置

### 5.8 论文模板与素材

论文模板只在管理员后台维护，平台统一使用当前启用模板，不对普通用户开放上传或选择。

MVP 建议至少支持：

1. 模板列表
2. 模板名称
3. 模板版本
4. 当前是否启用
5. 上传时间
6. 备注说明

管理员操作：

1. 上传模板
2. 替换模板版本
3. 设置当前启用模板
4. 下线错误模板
5. 查看当前全平台生效模板

### 5.9 系统通知

后台应支持给用户发系统级通知，避免每次都改前端提示。

MVP 支持：

1. 发布系统公告
2. 发布维护通知
3. 发布额度变更通知
4. 发布任务完成通知模板

### 5.10 操作日志

只要后台能改额度、改配置、改模型，就必须有日志。

至少要记录：

1. 谁操作的
2. 操作了什么
3. 改前值
4. 改后值
5. 操作时间
6. 关联用户或工作空间

为什么必须做：

1. 后面多人协作时，可以知道是谁改坏了配置。
2. 涉及额度补发和封禁时，必须可追溯。

## 6. 上线优先级

### 6.1 P0：公开测试前必须有

1. 概览看板
2. 用户管理
3. 工作空间管理
4. 任务中心
5. Token 与 AI 用量
6. 平台配置（只做基础配置）
7. 论文模板管理
8. 系统通知 / 公告
9. 操作日志

### 6.2 P1：第一批用户进来后尽快补齐

1. 订单与计费
2. 高消耗用户预警
3. 失败任务自动告警

### 6.3 P2：准备正式收费时补齐

1. key 池管理
2. 套餐配置
3. 充值与补单流水
4. 退款流程
5. 工单系统
6. 角色细分与权限点

## 7. 当前数据库可直接复用的数据

现有表已经能支撑后台第一版的大部分能力：

1. `User`
2. `Workspace`
3. `Order`
4. `UserQuota`
5. `AiUsageLog`
6. `TaskJob`
7. `Notification`

这意味着后台第一版不需要大改架构，优先做聚合查询和页面即可。

## 8. 建议新增的数据模型

为了让管理员端真正可运营，后续建议补以下几类表：

1. `AdminAuditLog`
   记录后台所有关键操作。
2. `SystemConfig`
   保存默认 token 配额、开关项、默认模型等全局配置。
3. `ModelProviderKey`
   保存不同模型供应商的 key、状态、优先级、失败次数。
4. `UserStatusLog`
   记录用户封禁、解封、补偿额度等历史。
5. `SupportTicket`
   后续如果要做问题反馈、人工介入，可接入工单。

## 9. 对当前项目的落地建议

结合当前阶段，我们按下面顺序推进，确保每一步都稳定可回归：

1. 先修管理员端现有对不齐问题，保证现有页面稳定可用。
2. 再做管理员端第一批 P0：工作空间管理 + 任务中心 + Token/AI 用量。
3. 论文模板只在管理员后台上传和启用，平台统一使用模板，不对普通用户开放上传或选择。
4. 最后补系统通知/公告与平台配置，形成完整运营闭环。

流程要求：

1. 每个功能点开发前先补对应 OpenSpec 变更说明。
2. 每个功能点开发并自测通过后立即归档 OpenSpec（archive）。

## 10. 一句话结论

管理员端除了“用户管理、token、模型/key 控制”之外，最缺的其实还有三块：

1. 工作空间管理
2. 任务中心
3. 操作日志与风控

这三块决定了平台出了问题时，你能不能快速定位、快速止损、快速恢复。
