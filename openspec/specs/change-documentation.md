# 变更文档规范

> 适用于所有功能开发、修改、修复操作

## 核心原则

每一次功能变更都必须在 `openspec/changes/` 中留下对应的文档记录，确保任何开发者接手项目后能快速理解"做了什么、为什么做、怎么做的"。

---

## 何时需要更新文档

| 场景 | 动作 |
|------|------|
| 新增功能模块 | 在 `changes/` 下新建模块目录，创建完整的 4 个文档 |
| 已有模块新增功能 | 更新对应模块的 `tasks.md`，必要时补充 `design.md` |
| Bug 修复 | 在对应模块的 `tasks.md` 底部"修复记录"区追加条目 |
| 跨模块变更 | 涉及的每个模块都需更新，或在最相关的模块中统一说明 |
| 重构 / 技术升级 | 更新对应模块的 `design.md`，在 `tasks.md` 追加迁移任务 |

---

## 模块目录结构

每个功能模块在 `openspec/changes/<module-name>/` 下包含以下文件：

```
changes/<module-name>/
├── proposal.md     # 需求提案
├── design.md       # 技术设计
├── specs.md        # 接口规格
└── tasks.md        # 任务清单
```

### proposal.md — 需求提案

回答"做什么"和"为什么做"。

```markdown
# <模块名称>

## 目标
一句话说明这个模块要解决什么问题。

## 优先级
高 / 中 / 低

## 背景
为什么需要这个功能，解决用户什么痛点。
```

### design.md — 技术设计

回答"怎么做"。

```markdown
# <模块名称> - 技术设计

## 方案概述
整体技术方案简述。

## 数据模型
涉及的数据库表 / 字段变更。

## API 设计
新增或修改的 API 端点、请求/响应格式。

## 组件设计
新增或修改的前端组件及其职责。

## 依赖
引入的第三方库或服务。
```

### specs.md — 接口规格

精确的接口定义，供前后端对齐。

```markdown
# <模块名称> - 接口规格

## API 端点

### POST /api/xxx
- 请求体: { field: type }
- 响应体: { field: type }
- 错误码: 400 / 401 / 404

## 类型定义
关键 TypeScript 接口。
```

### tasks.md — 任务清单

开发进度跟踪，每完成一项打勾。

```markdown
# <模块名称> - 任务列表

## 实现任务

- [x] T1: 任务描述
- [x] T2: 任务描述
- [ ] T3: 待完成任务

## 修复记录

- **修复**: 问题描述 → 解决方案
```

---

## 任务编号规则

- 格式：`T<序号>`，在模块内自增（T1, T2, T3...）
- 后续迭代的任务接着编号，不重置
- 可添加日期标注区分迭代批次：`## 后续迭代 (2026-03-16)`

---

## 修复记录格式

在 `tasks.md` 底部的"修复记录"区，每条修复包含：

```
- **修复**: <问题现象> → <根因和解决方案>
```

示例：
```
- **修复**: Worker 不消费任务 → API 路由缺少 taskQueue.add() 调用，新建 lib/queue.ts
```

---

## ROADMAP.md 同步

功能完成后，同步更新 `openspec/ROADMAP.md`：

1. 在"已完成模块"表中添加或更新条目
2. 从"剩余待办"中移除已完成项
3. 更新"文件统计"中的数字
4. 如有重要修复，追加到"关键修复记录"表

---

## 文档不是可选项

- 代码提交前确认文档已同步更新
- 没有文档的功能视为"未完成"
- 文档和代码同等重要，是项目可持续协作的基础
